# Proposal: agente-1-doc-query

> Generated by `sdd-propose` on 2026-06-27.
> Status: draft (awaiting user approval)

## Intent

Construir el primer agente Java del stack: `doc-query-agent`, un servicio que responde preguntas sobre documentación interna usando RAG sobre Qdrant. Inaugura el patrón de agente hexagonal Spring Boot + Spring AI que reutilizarán los próximos agentes del catálogo.

## Scope

- Un servicio Spring Boot 3.x / Java 21 corriendo en Docker, expuesto en `:8080`, integrado al `docker-compose.yaml` del stack.
- Endpoint `POST /api/v1/ingest` que procesa todos los archivos del directorio `/app/docs` (montado como volumen), los chunkea, genera embeddings y los persiste en Qdrant (colección `doc-query-docs`).
- Endpoint `POST /api/v1/query` que recibe `{ "question": "..." }`, recupera chunks relevantes, arma el prompt y devuelve `{ "answer": "...", "sources": [...] }`.
- Chat completion va via LiteLLM (`:4000`, alias `llama3`) — respeta la regla del stack.
- Embeddings van via LiteLLM (`:4000`, alias `nomic-embed-text`) — se agrega el modelo de embeddings a `litellm_config.yaml` apuntando a Ollama. Arquitectura consistente: un único gateway para todo el tráfico LLM.
- Arquitectura hexagonal: domain con ports, application con use cases, infrastructure con adapters Spring AI / REST / FileSystem.
- Privacidad: todo el flujo es local, ningún dato sale a internet.

## Out of scope

- Watcher de archivos en `/app/docs` (la ingesta es trigger-by-endpoint, no automática).
- Autenticación de los endpoints REST (queda para un change transversal cuando haya más de un agente).
- Reingesta incremental o detección de duplicados (la ingesta es full-refresh / append por ahora).
- Soporte de formatos más allá de PDF y Markdown.
- UI o cliente — solo backend REST.
- Integración con Langfuse (queda para un change de observabilidad cuando esté el agente andando).
- Otros agentes del catálogo (diagnosis, test-gen, spec-translator).

## Affected areas

- Nuevo módulo Maven: `agents/doc-query-agent/` (proyecto Spring Boot independiente).
- Modificación de `docker-compose.yaml`: nuevo service `doc-query-agent` con build context, volumen `./docs:/app/docs`, dependencia de `ollama`, `qdrant`, `litellm`.
- Nuevo directorio root `docs/` con `.gitkeep` para el volumen de ingesta.
- Sin cambios en `litellm_config.yaml` (el alias `llama3` ya existe).
- Sin cambios en Qdrant (la colección se crea on-demand desde el adapter).

## Rollback

Apagar el service en `docker-compose.yaml` (comentar el bloque o `docker compose stop doc-query-agent`) deshabilita el agente sin afectar al resto del stack. El módulo Maven y el directorio `docs/` quedan en el repo pero no se ejecutan. Para revertir totalmente: `git revert` del commit. La colección Qdrant `doc-query-docs` puede borrarse con un `DELETE` al endpoint de Qdrant — no hay datos compartidos.

## Success criteria

- `docker compose up doc-query-agent` levanta el servicio sin errores y queda escuchando en `:8080`.
- Con al menos un PDF/markdown en `./docs`, `curl POST /api/v1/ingest` devuelve 200 y un count de chunks > 0.
- `curl POST /api/v1/query` con una pregunta relacionada a los docs devuelve respuesta coherente + lista de sources no vacía.
- Una pregunta sin contexto en los docs devuelve una respuesta que admite no saber, sin alucinar.
- Los logs del container muestran llamadas a LiteLLM (chat) y a Ollama (embeddings) — confirma el ruteo.
- Las convenciones de `.sdd/conventions.md` (hexagonal, constructor injection, sin hardcodeo de modelo) se cumplen.

## Acknowledged risks

- Spring AI 1.x es reciente; las APIs (`ChatClient`, `VectorStore`, `EmbeddingModel`) pueden tener breaking changes entre minors. Mitigación: fijar versión exacta en el `pom.xml`.
- `nomic-embed-text` debe estar descargado en Ollama antes del primer `ingest`. Si no está, falla con error opaco. Documentar el `ollama pull` en el README del agente.
- Calidad de respuesta depende del chunking y del prompt; la primera versión va a ser tosca. Es aceptable para un MVP de aprendizaje.

## Technical Options Considered

- **Option A: Embeddings via LiteLLM**
  - When it fits: cuando querés un único punto de entrada para TODO el tráfico LLM (chat + embeddings), trazas unificadas en Langfuse y un solo lugar para cambiar de modelo.
  - Why not here / risks: el soporte de LiteLLM para embeddings de Ollama es flaky en 2026 (formato de respuesta inconsistente entre versiones). Agrega un punto de falla para una operación que es puramente local. La regla de "todo via LiteLLM" tiene sentido para chat (donde hay ruteo por privacidad y elección de modelo), pero embeddings con `nomic-embed-text` no tienen alternativa cloud razonable en este stack.

- **Option B: Embeddings directo a Ollama**
  - When it fits: cuando el modelo de embeddings es fijo, local y no necesita ruteo por privacidad. Simplifica config y elimina un hop.
  - Why not here / risks: rompe la consistencia arquitectónica ("todo va via LiteLLM"). El agente termina con dos clientes Spring AI configurados. Si en el futuro hay que cambiar de modelo de embeddings, hay que tocar código del agente, no solo `litellm_config.yaml`.

## Recommended Direction

- **Chosen option**: Option A — embeddings y chat ambos via LiteLLM.
- **Why**: decisión del usuario. Arquitectura consistente: un único gateway para todo el tráfico LLM. Se agrega `nomic-embed-text` a `litellm_config.yaml`. Spring AI usa `OpenAiEmbeddingModel` apuntando a `http://litellm:4000/v1/embeddings`.
- **Trade-off accepted**: si LiteLLM tiene comportamiento flaky con embeddings de Ollama, se detecta en verify y se puede caer a Option B sin tocar el dominio.

## Open decisions for next phases

- Estrategia de chunking concreta (tamaño, overlap) y splitter (`TokenTextSplitter` vs alternativos de Spring AI).
- Forma del prompt RAG: plantilla, cómo se inyectan los sources, manejo de "no sé".
- Política de ingesta: ¿`/api/v1/ingest` borra y recarga la colección, o hace append? Idempotencia.
- Manejo de errores de los endpoints (códigos HTTP, formato de error).
- Estructura de los `sources` devueltos: ¿path del archivo, snippet, score?

# Design: orchestrator-agent

> Generated by `sdd-design` on 2026-06-29.

## Summary

Nuevo microservicio Spring Boot hexagonal en `:8083` que expone `POST /api/v1/ask` y delega a uno de los tres agentes existentes vía LLM tool calling. Las tres `@Tool` (`callDocQueryAgent`, `callDiagnosisAgent`, `callSmartSearchAgent`) son HTTP POSTs sincrónicos con `RestClient` y registran el agente invocado en un `ThreadLocal<List<String>>`, replicando el patrón ya validado en `smart-search-agent`. El modelo es `gpt-4o-mini` (o `gemini-flash`) via LiteLLM; el ruteo lo decide el modelo según las descriptions de cada tool.

## Decisions

### D1: Config de URLs downstream — `@ConfigurationProperties` vs `@Value`

**Context**: el agente necesita tres URLs (`doc-query-agent`, `diagnosis-agent`, `smart-search-agent`) más timeouts.

**Options considered**:
1. **`@Value` campo a campo**: una propiedad por URL, inyectada directa en cada bean RestClient.
   - Pro: alineado con el baseline org (12-factor, sin defaults).
   - Con: 3 URLs + 2 timeouts dispersas, sin agrupación lógica.
2. **`AppProperties` con `@ConfigurationProperties(prefix="agents")`**: record con sub-records `docQueryAgent.url`, `diagnosisAgent.url`, `smartSearchAgent.url`.
   - Pro: agrupa configuración relacionada; tipa fuerte; la convención del proyecto lo permite explícitamente cuando hay "agrupaciones complejas con justificación".
   - Con: rompe con `@Value` puro del baseline.

**Decision**: `AppProperties` (Option 2).

**Rationale**: tres endpoints downstream + sus timeouts forman una agrupación coherente; tratarlos sueltos con `@Value` dispersa contrato de integración. `conventions.md §Agentes Java` lista `@ConfigurationProperties` con records como opción válida. Sin defaults inline: las env vars siguen siendo mandatorias (la app falla si faltan).

---

### D2: RestClient — un bean compartido vs uno por downstream

**Context**: tres downstreams con misma semántica pero distintas base URLs y mismos timeouts.

**Options considered**:
1. **Un `RestClient` global** con `baseUrl` por llamada.
   - Pro: menos beans.
   - Con: pierde la noción de "cliente del agente X"; cualquier cambio de timeout/policy afecta a todos.
2. **Tres `RestClient` beans** (`docQueryClient`, `diagnosisClient`, `smartSearchClient`), construidos en `SpringAiConfig` a partir de `AppProperties`.
   - Pro: cada tool inyecta SU cliente; encapsula `baseUrl` y timeouts por destino; trivial mockear por separado en tests.
   - Con: tres beans en vez de uno.

**Decision**: tres `RestClient` beans dedicados (Option 2).

**Rationale**: clarifica responsabilidades, evita pasar `baseUrl` por parámetro, y permite ajustar timeout de un downstream sin tocar el resto. Coste marginal (tres `@Bean` triviales) frente a evolución más limpia.

---

### D3: Aplastamiento de respuestas downstream

**Context**: cada downstream devuelve estructura distinta (`doc-query` → `{answer, sources}`, `diagnosis` → `{rootCause, location, suggestion}`, `smart-search` → `{answer, toolsUsed}`). Open decision del proposal.

**Options considered**:
1. **Reenviar JSON raw** al modelo como contexto de tool.
   - Pro: máxima información para que el modelo sintetice.
   - Con: el modelo recibe ruido (campos `sources`, `toolsUsed`) que infla tokens y puede contaminar la respuesta final.
2. **Aplanar a `String` en la tool** (extraer `answer` / concatenar `rootCause + suggestion`).
   - Pro: payload limpio al modelo; menos tokens; la spec del proposal ya lo define así.
   - Con: descarta metadata; si el usuario quisiera ver `sources`, no llega.

**Decision**: aplastar a `String` en cada tool (Option 2).

**Rationale**: el proposal acotó el contrato a `{answer, routedTo}`, sin metadata downstream. Reducir tokens importa con quota Gemini de 20 RPD (`learnings.md` 2026-06-29). Si más adelante hace falta exponer `sources`, se evoluciona el contrato.

---

### D4: Tool description para `callDiagnosisAgent`

**Context**: el modelo debe detectar que la pregunta contiene un log/stacktrace y pasarlo como parámetro `logContent`, no la pregunta entera. Riesgo identificado en proposal.

**Options considered**:
1. **Description genérica**: "diagnoses errors from logs".
   - Con: el modelo puede pasar la pregunta cruda; downstream falla o diagnostica basura.
2. **Description explícita y prescriptiva**: indicar (a) cuándo usarla (input contiene stacktrace, exception o log), (b) qué pasar como `logContent` (solo el bloque del log, no la pregunta envolvente), (c) ejemplo breve inline.
   - Pro: guía el ruteo y el shaping del parámetro.
   - Con: descriptions más largas consumen tokens del system prompt.

**Decision**: Option 2 — descriptions explícitas con regla de extracción.

**Rationale**: el riesgo está identificado en proposal y `learnings.md` confirma que los modelos pequeños son sensibles al wording. El coste en tokens es marginal frente a 502s por payload mal armado.

---

### D5: `routedTo` cuando el modelo no invoca ninguna tool

**Context**: si la pregunta es trivial ("hola") el modelo puede responder sin invocar tool. Open decision del proposal.

**Decision**: si `ThreadLocal` queda vacío, `routedTo` debe retornar `null` (o estar ausente en el DTO de respuesta). El contrato sigue devolviendo 200 con `answer` del modelo.

**Rationale**: evita romper el contrato y refleja con honestidad qué pasó, alineando el diseño con la especificación de la API para el DTO.

---

### D6: Mapeo de fallos downstream → HTTP

**Decision**: `RestClient` configurado con `defaultStatusHandler` que lanza `DownstreamAgentException` (runtime) ante 4xx/5xx o timeout; `GlobalExceptionHandler` la mapea a HTTP `502 Bad Gateway` con `ErrorResponse{error, message}`.

**Rationale**: 502 expresa correctamente "upstream failure" y es lo que el proposal acordó. Una excepción dedicada evita filtrar `RestClientException` a través de capas.

## New component shapes

### `OrchestratorController` (infrastructure/adapter/rest)
**Responsibility**: expone `POST /api/v1/ask`.
**Public contract**:
```java
ResponseEntity<AskResponseDto> ask(@Valid @RequestBody AskRequestDto request);
```
**Dependencies**: `OrchestrateUseCase`.

### `OrchestrateUseCase` (application/usecase)
**Public contract**:
```java
OrchestrationResult orchestrate(OrchestrationRequest request);
```
**Dependencies**: `OrchestrationPort`.

### `OrchestrationPort` (domain/port)
```java
OrchestrationResult orchestrate(OrchestrationRequest request);
```

### `SpringAiOrchestrationAdapter` (infrastructure/adapter/ai) implements `OrchestrationPort`
**Dependencies**: `ChatClient`, `AgentTools`.

### `AgentTools` (infrastructure/adapter/ai) — singleton
```java
@Tool(description="…") String callDocQueryAgent(String query);
@Tool(description="…") String callDiagnosisAgent(String logContent);
@Tool(description="…") String callSmartSearchAgent(String question);
void clearRoutedTo();
List<String> getRoutedTo();
```
**Dependencies**: tres `RestClient` (`docQueryClient`, `diagnosisClient`, `smartSearchClient`).

### `AppProperties` (infrastructure/config)
```java
@ConfigurationProperties(prefix="agents")
record AppProperties(AgentConfig docQueryAgent, AgentConfig diagnosisAgent,
                     AgentConfig smartSearchAgent, Duration connectTimeout, Duration readTimeout) {
  record AgentConfig(String url) {}
}
```

### `SpringAiConfig` (infrastructure/config)
**Beans**: `ChatClient`, `RestClient docQueryClient`, `RestClient diagnosisClient`, `RestClient smartSearchClient`.

### Domain records
`OrchestrationRequest(String question)`, `OrchestrationResult(String answer, String routedTo)`.

### DTOs
`AskRequestDto(@NotBlank String question)`, `AskResponseDto(String answer, String routedTo)`, `ErrorResponse(String error, String message)`, `DownstreamAgentException`.

## Cross-cutting concerns

### Timeouts y resiliencia
`RestClient` configurado con `connectTimeout=5s`, `readTimeout=30s`. `spring.ai.retry.max-attempts=1` para no quemar quota Gemini.

### Observability
Log a INFO al entrar al use case (sin la pregunta completa, solo longitud) y al invocar cada tool (`"routing to <agent>"`). Log a ERROR en `DownstreamAgentException` con status downstream. Nunca loguear `answer` ni `logContent` (puede contener PII).

### Backward compatibility
Cambio aditivo puro. Ningún agente existente cambia.

## Applied design principles

- **Ports & Adapters**: el LLM y los HTTP calls a downstream son adapters de salida detrás de `OrchestrationPort`. Permite testear el use case mockeando un solo port. No aplica si en el futuro el ruteo deja de involucrar LLM (ahí cambia el port).
- **Open/Closed sobre tools**: sumar un cuarto agente = nueva `@Tool` + nuevo `RestClient` bean. No se toca `OrchestrateUseCase` ni el controller.

## Deferred decisions

- **Formato de error**: simple `ErrorResponse` ahora; Problem Details (RFC 7807) cuando haya más de un consumer del API.
- **Timeout configurable por downstream**: por ahora valores globales en `AppProperties`; si un agente necesita timeout distinto, se parte el record.
